7. FLUXO ATUAL DO CODIGO

1. O utilizador submete uma notificacao pelo Vaadin ou pela API REST.
2. NotificationService cria um NotificationJob com status PENDING.
3. O job e gravado na tabela notification_jobs.
4. O jobId real e colocado dentro do NotificationRequest.
5. O pedido e enviado para RabbitMQ.
6. RabbitMQ encaminha a mensagem para EMAIL, SMS ou PUSH.
7. NotificationConsumer processa a mensagem.
8. O consumer simula o envio com delay de 500ms.
9. O consumer grava NotificationLog com o jobId original.
10. O consumer atualiza NotificationJob para SUCCESS ou FAILED.
11. Se existir callbackUrl, o sistema envia um callback HTTP assincrono.

Nota:
Este fluxo atualizado ja esta no codigo, mas a aplicacao que estava a correr na
porta 8080 foi iniciada antes das alteracoes. Para testar o novo comportamento,
e necessario reiniciar a aplicacao.


8. API REST

Endpoint para criar notificacao:
POST /api/v1/notifications

Exemplo de JSON:

{
  "clientId": "cliente-1",
  "channelType": "EMAIL",
  "recipient": "teste@exemplo.com",
  "templateName": "template_teste",
  "parameters": {
    "texto_mensagem": "Mensagem de teste"
  },
  "callbackUrl": "https://sistema-origem.exemplo.com/webhook"
}

Resposta esperada:

{
  "jobId": "uuid-gerado",
  "status": "PENDING"
}

Endpoint para consultar estado:
GET /api/v1/notifications/{jobId}


9. CALLBACK / WEBHOOK

Foi adicionado suporte inicial a callbackUrl.

Quando o consumer termina o processamento, envia um HTTP POST assincrono para a
callbackUrl, se esta existir.

Payload enviado no callback:
- jobId
- status
- channelType
- recipient
- errorMessage
